    使用方式

    编译参数： -fsanitize=address：核心开关，开启编译器插桩和运行时库替换。 -g：极其重要。生成调试信息，这样报错时才能看到具体的“文件名:行号”，而不是一堆冰冷的十六进制内存地址。 -fno-omit-frame-pointer：为了获得更完美的调用栈（Call Stack），建议加上此参数，禁止编译器优化掉帧指针。 -O1：建议开启基础优化。在 -O0 下，有些过于简单的越界可能会被编译器直接在前端规避掉，反而测不出来。（-O1 这里有坑，如果错误太简单，编译器会做死代码优化）

    内存的映射和插桩

    AddressSanitizer主要包括两部分： 1. 插桩(Instrumentation) 2. 动态运行库(Run-time library)。

    插桩主要是针对在llvm编译器级别对访问内存的操作(store，load，alloca等)，将它们进行处理。

    动态运行库主要提供一些运行时的复杂的功能(比如poison/unpoison shadow memory)以及将malloc,free等系统调用函数hook住。

    该算法的思路是：如果想防住Buffer Overflow漏洞，只需要在每块内存区域右端（或两端，能防overflow和underflow）加一块区域（RedZone），使RedZone的区域的影子内存（Shadow Memory)设置为不可写即可。具体的示意图如下图所示。

    虚拟地址空间被分为两个不互相重叠的区域：

    • 主应用内存（Mem）：这部分内存由常规的应用程序代码使用；
    • 影子内存（Shadow）：这部分内存用于存放影子值（或成为元数据）。影子内存与主应用内存之间存在严格的映射对应关系。对主内存中的某个字节进行毒化，实际上意味着向其对应的影子内存中写入一个特定的标志值。

    这两类内存的组织方式必须确保一个前提：从主内存地址计算出对应的影子内存地址（即 MemToShadow 转换）必须非常快速。

    为什么叫“影子内存 (Shadow Memory)”？

    在 ASan 的世界里，主应用内存中的每 8 个字节（Byte），都对应着影子内存中的 1 个字节。 影子内存就像是主内存的“状态监视器”。应用程序只管在主内存里读写数据，而 ASan 则在影子内存里暗中记录这 8 个字节是“健康可用”的，还是被“毒化（越界或已释放）”的。

    “毒化 (Poisoning)”的具体动作是什么？

    当你在 C++ 中调用 delete 或 free() 时，或者系统为你分配红区（Red zones）时，ASan 会自动找到对应的影子内存，往里面写入特殊的值（也就是所谓的特殊标志值）：

    • 比如写入 0xFA 代表这里是堆内存的越界红区。
    • 写入 0xFD 代表这块内存已经被释放（Quarantine 状态）。 这就完成了“毒化”。

    为什么强调 MemToShadow 必须“非常快速”？

    因为每一次你的 C++ 代码尝试读取或写入内存（比如 *ptr = 10;），被“插桩”后的程序都要先去影子内存里查一下 ptr 的状态。

    如果计算地址的过程很慢，程序的性能就会遭受毁灭性打击。

    为了做到极致的快，ASan 的 MemToShadow 算法通常极其简单，利用了位运算： ShadowAddr = (MemAddr≫3) + Offset

    • MemAddr ≫ 3：把主内存地址除以 8（右移 3 位），因为 8 字节对应 1 字节影子内存。
    • Offset：加上一个固定的基准偏移量，直接跳到影子内存区域。

    整个转换过程只需要两条 CPU 汇编指令，这就是它能在生产环境中被广泛使用的底气。

    映射（Mapping）

    AddressSanitizer 将应用程序中每 8 个字节的内存映射为影子内存中的 1 个字节。 对于应用程序中任意按 8 字节对齐的内存块（qword），其状态只可能出现 9 种不同的取值： 1. qword 中的所有 8 个字节都未被毒化（即全部合法可寻址）。对应的影子值为 0。 2. qword 中的所有 8 个字节都被毒化（即全部不可寻址）。对应的影子值为负数。 3. 前 k 个字节未被毒化，剩余的 8 − k 个字节被毒化。对应的影子值为 k。 第 3 种情况之所以成立，是因为 malloc 总是返回按 8 字节对齐的内存块。在一个按 8 字节对齐的 qword 中，出现不同状态字节的唯一情况，就是 malloc 分配区域的尾部。

    例如，如果我们调用 malloc(13)，我们将得到一个完整的、未被毒化的 qword（8 字节），以及一个只有前 5 个字节未被毒化的 qword（8 + 5 = 13）。

    假设我们分配了 13 字节，基地址为 0x1000（这是一个 8 的倍数）。 - 第一个 8 字节 (0x1000 - 0x1007)：全安全，影子值为 0。 - 第二个 8 字节 (0x1008 - 0x100F)：只有前 5 个字节安全，影子值为 5。

    场景 A：合法读取尾部 你在 C++ 中写了：char c = ptr[12]; （读取第 13 个字节，合法） - address = 0x100C。 - shadow_value = 5。 - kAccessSize = 1 (char 占 1 字节)。 - 计算 (address & 7)：0x100C & 7 = 4 （这是该地址在当前 8 字节块内的偏移量）。 - 计算 last_accessed_byte：4 + 1 - 1 = 4。 - 判断：4 >= 5 为 False。安全，放行！

    场景 B：发生尾部越界 (Tail Out-of-bounds) 你在 C++ 中写了：int* p = (int*)(ptr + 10); int val = *p; （从第 11 个字节开始读取 4 个字节的大块，这会读到第 14 个字节，越界！） - address = 0x100A。 - shadow_value = 5。 - kAccessSize = 4 (int 占 4 字节)。 - 计算 (address & 7)：0x100A & 7 = 2。 - 计算 last_accessed_byte：2 + 4 - 1 = 5。 - 判断：5 >= 5 为 True。越界，立即报错拦截！

    MemToShadow(ShadowAddr) 的计算结果会落入 ShadowGap（影子间隙）区域，该区域是不可寻址的。因此，如果程序试图直接访问影子区域的内存，它将会崩溃。

    无论是在 64 位还是 32 位系统，ASan 都使用同一个极其精简的数学模型： Shadow = (Mem≫3) + Offset - Mem：是你 C++ 代码中实际访问的指针地址（比如 0x000012345678）。 -  ≫ 3：右移 3 位，等价于除以 8。这正是我们之前讲过的“8 字节主内存映射为 1 字节影子内存”在代码级别的实现。 - Offset：系统预留的影子内存起始基址（Base Address）。 - 64 位下是 0x7fff8000 - 32 位下是 0x20000000

    为什么这样设计？ 因为位移（Shift）和加法（Add）是 CPU 执行最快的两条指令。每次指针读写前，CPU 只需要 1 个时钟周期就能算出对应的影子地址，这是 ASan 性能损耗极低的核心秘密。

    内存空间的“三明治”布局:

    由于虚拟地址空间非常庞大，ASan 不能简单地把内存切成两半。它采用了类似“三明治”的五层布局，将应用程序代码和影子内存交织在一起。

    以 64 位系统 的布局为例进行剖析： #### 🍞 第一层和第五层：主应用内存 (Mem) - [0x000000000000, 0x00007fff7fff] LowMem (低位主内存) - [0x10007fff8000, 0x7fffffffffff] HighMem (高位主内存) - 用途：这是你的 C++ 应用程序真正生活的地方。你的代码指令、堆区 (malloc/new)、栈区局部变量，全都被操作系统分配在这两块区域里。

    🥬 第二层和第四层：影子内存 (Shadow)

    • [0x00007fff8000, 0x00008fff6fff] LowShadow (低位影子内存)
    • [0x02008fff7000, 0x10007fff7fff] HighShadow (高位影子内存)
    • 用途：这里存放着所有的“毒化”状态。
    • 映射关系：LowMem 中所有数据的状态记录在 LowShadow 里；HighMem 中所有数据的状态记录在 HighShadow 里。

    🥩 最核心的中间层：影子间隙 (ShadowGap)

    • [0x00008fff7000, 0x02008fff6fff] ShadowGap
    • 用途：这是一片绝对的“死区（Unmapped/Protected）”，操作系统通过页表控制，任何对这里的读写都会立刻引发段错误（Segfault）并让程序崩溃。
    • 为什么需要死区？ 前文提到过，如果黑客或者野指针企图直接篡改影子内存的值，或者由于某种逻辑错误，程序尝试对“影子内存自身的地址”代入公式计算其“影子的影子”，算出来的结果必然会精准落在这个 ShadowGap 区间内。这就构成了一个完美的物理隔离带，防止 ASan 的底层安全机制被破坏。

    为什么在 return 前要“解毒”？

    这是栈溢出检测和堆溢出检测最大的区别。 - 堆内存（Heap） 被 free 之后，直接丢进隔离区，毒药永远留着。 - 栈内存（Stack） 的生命周期随函数结束而销毁。当 foo() 函数执行 return 后，它的栈帧（Stack Frame）会被操作系统回收，并立刻分配给下一个被调用的函数使用。 - 如果不把刚刚布置的 0xff 毒药全部清零（恢复为安全的 0），那么下一个正常使用这块栈空间的函数就会莫名其妙地触发 ASan 报错。这被称为 “打扫战场” 。

    当你的 C++ 代码是 long value = *a;（读取完整的 8 字节）时，由于 8 字节正好对应 1 个完整的影子字节，检查逻辑极其简单：只要影子字节不是 0，就说明有错。

    运行时动态库 （Run-time library)

    Malloc

    运行时库替换了标准的 malloc 和 free 函数，并提供了类似 __asan_report_load8 这样的错误报告函数。

    malloc 的改造：当分配请求指定大小的内存时，它会在该内存周围额外分配红区（redzones）。与红区对应的影子内存会被毒化（poisoned），而分配给用户的主内存区域的影子值会被清零（标记为安全可用）。

    free 的改造：它会将整块内存区域的影子值全部毒化，并将这块内存放入隔离队列（quarantine queue）中（以此确保在接下来的一段时间内，这块内存绝不会被 malloc 再次分配出去）。

    它是如何防范漏洞的？

    ASan 的运行时库通常通过预加载（预链接）技术，在程序启动时抢先接管系统的动态内存分配大权。你以为你在调用底层的 libc，其实你调用的是 ASan 的“伪装者”。

    内存泄漏检测 - AddressSanitizerLeakSanitizer

    1. 核心原理：停机清点与可达性分析

    LeakSanitizer 的检测时机是在程序即将退出时（通常是挂载在 atexit 钩子上）。它采用了一种类似 Java/Go 垃圾回收器（GC）中的“标记-清除（Mark-and-Sweep）”算法的变体——可达性分析（Reachability Analysis）。

    它的工作流程分为三个阶段： #### 阶段 A：暗中记账 (Tracking) 从程序启动开始，ASan 接管了每一次 malloc 和 new。它在后台悄悄维护着一张“全量活跃内存表”，记录着当前所有已分配但尚未 free/delete 的堆内存块。

    阶段 B：寻找根节点 (Root Set Scanning)

    当程序执行到 return 0; 准备退出时，LSan 会“暂停世界（Stop-The-World）”，开始搜查系统中所有的“根集合（Root Set）”。根集合包括： 1. 全局变量数据段 (.data 和 .bss)：所有的全局指针。 2. 当前所有线程的栈 (Thread Stacks)：所有还活着的局部指针变量。 3. CPU 寄存器 (Registers)：可能临时存放着指针的寄存器。

    阶段 C：顺藤摸瓜与定罪 (Marking & Reporting)

    • 标记： LSan 把“根集合”当作起点，顺着里面的指针，去扫描整个堆内存。如果堆上一块内存被根集合里的指针指着，或者被另一块“合法可达”的堆内存指着，LSan 就认为它是“活着的（Reachable）”。
    • 定罪： 遍历结束后，LSan 会拿出阶段 A 记下的“全量活跃内存表”进行对账。那些没有被任何有效指针指向的内存块，就会被判定为“内存泄漏（Leaked）”。因为在 C++ 中，如果你连指针都弄丢了，这块内存就永远不可能被释放了。
